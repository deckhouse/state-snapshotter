**N2b — generic [`spec/system-spec.md`](../spec/system-spec.md) §3 в коде:** **E1–E5** — как в таблице и [`design/implementation-plan.md`](../design/implementation-plan.md) **§3-E1–E5**. **E5:** root own MCR содержит только namespace-scoped allowlist objects и исключает объекты из descendant MCPs, включая common SnapshotContent nodes; Kubernetes **Namespace** object не захватывается. Если own scope пустой, создаётся empty MCP (0 objects), и `manifestCheckpointName` всё равно заполняется. **Children readiness:** parent content `Ready` по строгим refs + **`Get`** по GVK; snapshot `Ready` только зеркалит bound content. Registry только для **E5** / graph walk при непустых **`childrenSnapshotRefs`** на exclude-пути. Дерево run **namespace-local** (**§3.2** spec): форма ref — **`apiVersion/kind/name`** (без `namespace`), child namespace всегда берётся от parent. Parent controllers владеют graph edges: `Snapshot` пишет top-level refs из CSD discovery, VM пишет refs на свои Disk children; child controllers не self-register. Кластерный smoke registry — **`hack/snapshot-graph-registry-smoke.sh`**.

**Always-on runtime:** `Snapshot`, common `SnapshotContent`, CSD reconciler, graph registry, `GenericSnapshotBinderController` / `unifiedruntime.Syncer`, and dynamic CSD hot-add path initialize on the single v0 runtime path.

**Domain capture SDK (`pkg/snapshotsdk`) — ✅ v1 capture-only:** standalone Go module (own `go.mod`, consumed via `replace`) that hides the capture-side Kubernetes transport (ManifestCaptureRequest, storage-foundation VolumeCaptureRequest, owner references, optimistic-locked status patches, the `PlanningReady` planning barrier, orphan child GC) behind intent verbs `EnsureChildren` / `EnsureVolumeCapture` / `EnsureManifestCapture` / `MarkPlanningReady` / `MarkPlanningFailed` / `MarkNotReady`. The demo `domain-controller` snapshot reconcilers are migrated onto it via a `SnapshotAdapter` seam (no behavior change; domain keeps only source validation, PVC resolution, and child-object building). The restore extension point moved to `pkg/snapshotsdk/transform` (old `domain-controller/pkg/domainsdk` and `pkg/volumecapture` removed). Design: [`design/domain-sdk-plan.md`](../design/domain-sdk-plan.md), tests [`design/domain-sdk-test-plan.md`](../design/domain-sdk-test-plan.md).

**CSD-gated demo activation:** graph registry built-in по умолчанию содержит только `Snapshot`→ common content. Demo VM/Disk resources входят в `Snapshot` tree только через eligible CSD. Hot-add CSD покрыт для новых root snapshots; requeue уже существующих root после активации CSD остаётся follow-up при необходимости.

**Common `SnapshotContent` ownership:** `SnapshotContent` is now retained/self-contained: it has no live reverse reference to Snapshot, its spec is immutable after creation, and its durable result graph lives in status. Snapshot-domain controllers own planning/execution requests, write their own `XxxxSnapshot.status` (`boundSnapshotContentName`, request names, `childrenSnapshotRefs`, `PlanningReady`, mirrored snapshot `Ready`), and publish result refs into bound `SnapshotContent.status` (`manifestCheckpointName`, N5 target `dataRefs[]`, `childrenSnapshotContentRefs`). `SnapshotContentController` validates those persisted refs, ensures artifact ownerRef handoff, and owns only content readiness conditions. MCR/VCR `Ready=True` means the artifact is ready and owned by `SnapshotContent`; cleanup request runs only after that explicit chain. Published data refs point to cluster-scoped final artifacts (`apiVersion`/`kind`/`name`; v0 target — `VolumeSnapshotContent`), not execution requests such as VCR or DataExport.

**Manifest payload security:** `ManifestCheckpointContentChunk` RBAC is internal-only: module templates grant chunks only to the controller/API service account with `create/get/delete`; user-facing read access is through `/manifests`; the API assembles payloads by reading chunks via the internal direct client, while direct chunk `get/list/watch` is not granted to users.

**Condition model (PR2a–PR2c; update 2026-06-14):** публичная модель — 5 условий `PlanningReady` (gen-gated gate/барьер), `ManifestsReady`, `VolumesReady`, `ChildContentsReady`, `Ready`. `SnapshotContent.Ready = ManifestsReady && VolumesReady && ChildContentsReady` (единственный агрегатор); `Snapshot.Ready` зеркалит content (исключение — bridge терминального child-Snapshot capture failure). Failure листа поднимается по ancestor-chain как `ChildrenFailed`, sibling-ветки не затрагиваются (INV-FAIL1). `RequestsReady` разнесён на `ManifestsReady` + `VolumesReady` ради ops-диагностики (см. ADR §2.2); это новые условия, не возврат удалённых в PR2c одноимённых мёртвых констант. Снятые условия: `GraphReady`, `Bound` (→ поле `status.boundSnapshotContentName`), `InProgress`, `HandledByCustomSnapshotController`, `HandledByCommonController`, `RequestsReady`. Барьер binder — `PlanningReady=True` при текущей generation. Детали: [`spec/system-spec.md` §3.8/§3.9.7](../spec/system-spec.md), ADR `snapshot-rework/2026-06-03-snapshot-conditions-model.md`.

**Lifecycle ownerRef model:** any Snapshot kind can be a root run, including standalone demo Snapshot kinds. Root `SnapshotContent` has `ownerRef -> root ObjectKeeper`. Root ObjectKeeper follows root Snapshot via `spec.followObjectRef`. Root Snapshot itself is not owned by ObjectKeeper because Snapshot is namespaced and ObjectKeeper is cluster-scoped. Child Snapshot ownerRef points to parent Snapshot; child `SnapshotContent` ownerRef points to parent `SnapshotContent`; MCP / future data artifacts ownerRef points to the owning `SnapshotContent`. `SnapshotContent` must not be owned by short-lived Snapshot and must not be ownerless after reconcile convergence.

**Same-name root recreate:** retained root `ObjectKeeper` identity is bound to one snapshot UID through `ObjectKeeper.spec.followObjectRef.UID`. Same namespace/name root Snapshot cannot reuse retained root ObjectKeeper if that ObjectKeeper follows an old Snapshot UID. A new root Snapshot with the same namespace/name must fail closed or wait until old root ObjectKeeper expires/is removed.

**Orphan PVC (root residual via CSI VolumeSnapshot) — ✅ capture-side в коде + e2e:** capture-side слайс в треке N5/namespace-flow. Для **orphan/uncovered PVC** root-узел не создаёт VCR, а использует стандартный CSI `VolumeSnapshot`; bound `VSC` публикуется в root `SnapshotContent.status.dataRefs[]` (durable, `deletionPolicy=Retain`, ownerRef→content), `VolumeSnapshot` виден как `Snapshot.status.childrenSnapshotRefs` leaf (visibility) и **не** попадает в MCP/inventory; PVC-манифест остаётся в MCP. VCR — только для domain-узлов. Контракт §3.9.11; ADR [`snapshot-rework/2026-06-09-orphan-pvc-csi-volumesnapshot.md`](../../../snapshot-rework/2026-06-09-orphan-pvc-csi-volumesnapshot.md).
