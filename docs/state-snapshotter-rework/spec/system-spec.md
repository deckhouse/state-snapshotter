Полная детализация CSD / registry / RBAC — в ADR [`snapshot-rework/2026-01-23-unified-snapshots-registry.md`](../../../snapshot-rework/2026-01-23-unified-snapshots-registry.md); не дублировать длинные фрагменты здесь без необходимости — обновлять этот файл при изменении контракта.

Нумерация разделов ниже совпадает с бывшим указателем `snapshot-rework/plan/dorabotki-i-testy.md`: **§0** — registry/runtime, **§1** — контекст, **§2** — ссылки, **§3** — граф snapshot-run (PR5+).

## §0. Registry state и runtime (watch)

Две подсистемы:

| Подсистема | Роль | Динамичность |
|------------|------|----------------|
| **Generic unified runtime state** | Желаемый набор типов generic `Snapshot` / `SnapshotContent` wiring из runtime bootstrap + eligible CSD + discovery GVK/GVR | Always-on CSD reconciler + `pkg/unifiedruntime.BuildLayeredGVKState` (eligible → merge → resolve); снимок в `Syncer.lastState` |
| **Graph registry state** | Пары `XxxxSnapshot` / `SnapshotContent`, доступные для `Snapshot` discovery и generic graph traversal | Встроенный минимум только **`Snapshot` → `SnapshotContent`** + eligible CSD + `RESTMapper`; demo VM/Disk **не** активируются по умолчанию |
| **Runtime watch activation** | Подписки controller-runtime на `*Snapshot` / `*SnapshotContent` | Quasi-dynamic; **снятие watch без рестарта** не гарантируется. Для **`Snapshot`** — отдельные **additive** watches на дочерние snapshot-GVK из живого graph registry (регистрация при reconcile); parent просыпается от событий дочернего объекта через map **child → parents** по **`apiVersion`/`kind`/`name`** в **`status.childrenSnapshotRefs`** (поиск родителей только в namespace дочернего объекта — **namespace-local** дерево run, см. **§3.2**) |

**Правило:** отключение или удаление типа может потребовать **рестарта pod** для консистентного cleanup watch. См. ADR *Registry state vs runtime*.

## §1. Контекст продукта (кратко)

- **S1–S2:** optional unified CRD могут отсутствовать; процесс не падает. Пары GVK отфильтровываются через `pkg/unifiedbootstrap` + `RESTMapper` (см. код и план в design/testing).
- **R1 ✅:** типы и CRD **`CustomSnapshotDefinition`** (`api/v1alpha1`, `crds/`).
- **R2 phase 1 ✅:** reconciler в manager (`internal/controllers/csd/controller.go`): resolve маппинга по CRD, `Accepted` / производный `Ready`, `KindConflict` и `InvalidSpec` (в т.ч. namespace-scoped content); `AccessGranted` пишет только hook.
- **R2 phase 2a ✅:** на старте процесса merge eligible CSD ∪ bootstrap → resolve mapper → начальные GVK для unified контроллеров (`cmd/main.go`, `pkg/csdregistry`, `pkg/unifiedbootstrap`).
- **R2 phase 2b ✅:** после успешного reconcile CSD — `pkg/unifiedruntime.Syncer.Sync`: пересчёт слоёв (`LayeredGVKState`) и additive `AddWatch*` для новых resolved пар; **clean unwatch не гарантируется** (ADR).
- **Always-on runtime startup (v0):** `Snapshot`, common `SnapshotContent`, `DemoVirtualDiskSnapshot`, `DemoVirtualMachineSnapshot`, CSD reconciler, graph registry, `GenericSnapshotBinderController`, `unifiedruntime.Syncer`, and dynamic hot-add watches are initialized on the single runtime path. CSD управляет не фактом запуска этих controllers, а тем, входит ли domain kind в **graph registry** и `Snapshot` discovery.
- **Common content model (v0 → target):** `storage.deckhouse.io/SnapshotContent` is the only active cluster-scoped content carrier and has no live reverse reference to the source snapshot. It represents a **logical snapshot node** (not one volume). It exposes immutable `spec` (`deletionPolicy`), publisher-owned `status.manifestCheckpointName` (MCP with many manifest targets via MCR), `status.childrenSnapshotContentRefs`, and a durable data binding: **`status.dataRef`** (singular, ≤1; per-PVC target identity + cluster-scoped artifact ref). Validator-owned `status.conditions`. CSD mapping uses only explicit GVK refs (`source.apiVersion`, `source.kind`, `snapshot.apiVersion`, `snapshot.kind`) plus `priority`; `resourceCRDName` / `snapshotCRDName` are not part of the active contract. The content side is fixed to common `SnapshotContent`. Snapshot-domain controllers own planning/execution and publish result refs into bound content status. `SnapshotContentController` validates persisted content refs and owns only content readiness conditions; it does **not** read live Snapshot objects, does **not** read MCR/VCR requests, and does **not** create execution requests. Snapshot-level `Ready` mirrors bound content `Ready`. Published data artifact refs MUST NOT point to `VolumeCaptureRequest`, `DataExport`, or other execution requests; first expected artifact kind is cluster-scoped `VolumeSnapshotContent` or equivalent. **`dataRefs[]` is present in CRD since PR-1**; validation, publish, and readiness paths are delivered by **PR-2+** / **PR-4** (see **§3.9**).
- **Domain controller barrier (v0):** `PlanningReady` MUST be published **only** by a domain/namespace controller (the actor that plans the snapshot). The common layer — including `GenericSnapshotBinderController` and the shared binding helpers — MUST NOT self-publish `PlanningReady`; it only **waits** for it. The owning domain controller MUST set `status.conditions[type=PlanningReady]=True` with `observedGeneration == metadata.generation` (gen-gated, **INV-COND3**) on its `XxxxSnapshot` only **after** planning for the current generation is complete: all required child snapshots are planned/published **and** all required own requests (MCR/VCR) are created/published (e.g. a leaf domain snapshot publishes `PlanningReady=True` only after its own MCR is planned, not on bind). `GenericSnapshotBinderController` MUST wait for `PlanningReady=True` at current generation before binding/creating `SnapshotContent`. `PlanningReady` — единственное barrier-condition; любые прежние имена barrier-condition сняты и MUST NOT использоваться активными контроллерами или тестами.
- **Request → artifact → content lifecycle:** snapshot controllers create MCR/VCR-style execution requests, set ownerRef → owning snapshot, store temporary request names in snapshot status, and publish durable result refs to the bound `SnapshotContent`. Request controllers create MCP/data artifacts and protect them through ObjectKeeper until handoff. `SnapshotContentController` validates the final artifacts and ensures artifact ownerRef → `SnapshotContent`. Snapshot controllers may clear request names and delete requests only after the explicit artifact chain is complete: content status references the artifact, the artifact exists, is `Ready=True`, and is owned by that `SnapshotContent`.
- **Snapshot lifecycle ownerRef model:** every non-root `XxxxSnapshot` and every `SnapshotContent` has exactly one lifecycle owner. Root `SnapshotContent` has `ownerRef -> root ObjectKeeper`. A root snapshot-run is anchored by one root `ObjectKeeper` in `FollowObjectWithTTL` mode: the root Snapshot is the ObjectKeeper follow target via `spec.followObjectRef`, but is not owned by that cluster-scoped ObjectKeeper because Snapshot is namespaced. Child Snapshot ownerRef points to parent Snapshot; child `SnapshotContent` ownerRef points to parent `SnapshotContent`; artifacts referenced from `SnapshotContent.status` are owned by that `SnapshotContent`. `SnapshotContent` MUST NOT be owned by a short-lived Snapshot.
- **R3 ✅ (ядро):** явный слой state в `pkg/unifiedruntime`; интеграционный proof hot-add — `test/integration/unified_runtime_hot_add_test.go`; Prometheus gauges + лог при «stale» active (ключ есть в monotonic active, но выпал из resolved). **Опционально:** доп. proof-сценарии — по плану.
- **Цель (ядро):** регистрация типов через CSD + **AccessGranted** + активация watch без рестарта для новых eligible типов — реализовано для additive-пути; симметричное снятие watch — нет.
- **Manifest / MCR / ManifestCheckpoint** — отдельный трек от unified registry snapshot-типов; не смешивать с CSD.
- **ManifestCheckpointContentChunk RBAC (US-18.1):** chunk CRs contain persisted manifest payload data and MUST NOT be exposed to ordinary users through direct Kubernetes API access. Module RBAC MUST grant `manifestcheckpointcontentchunks` only to the internal controller/API service account, with verbs `create`, `get`, `delete` and without `list` / `watch`. User-facing access to manifest payloads goes through the controlled `/manifests` API path, which reads chunks using the internal controller/API direct client under the controller/API pod service account identity; no user impersonation is used for chunk reads.
- **Raw capture policy (источник истины — MCP):** snapshot-capture stores manifests **as-is** in `ManifestCheckpoint`, **including `status`** and runtime fields; MCP is the source of truth for import/export (`DataImport`/`DataExport` read `status.capacity`, `spec.storageClassName`, `spec.volumeMode`, etc. from the original manifest). **Object selection** (`ShouldIncludeNamespaceObject`: skipping controller-owned dependents, control-plane noise, snapshot/own machinery) is a separate capture layer that does not mutate fields. **Field-level sanitization** (`status`, `metadata.managedFields`, `resourceVersion`, `uid`, `creationTimestamp`, …) happens **only** on the restore read-path (`internal/usecase/restore`), independent of capture. There are **no** field-level exceptions on capture (including `Secret` bytes — see US-18.2).
- **ManifestCheckpoint Secret handling (US-18.2):** Kubernetes `Secret` objects are captured **verbatim** like every other object — `.data` / `.stringData` are stored as-is, with no secret-specific selection or field stripping on the capture path. Secrets excluded by the unified inclusion rule (e.g. `kubernetes.io/service-account-token`) are not captured at all; this is object selection, not field stripping. At-rest encryption of the snapshot store is a separate, future concern; the legacy opt-in annotations (`include-secret`, `include-secret-data`) and secure-by-default stripping have been removed.
- **Logical SnapshotContent + capture (N5+, Variant A target):** **`SnapshotContent`** = logical node (MCP + singular **`status.dataRef`** ≤1 + children = domain decomposition **and** child volume nodes); a domain leaf owns one PVC (own MCR + VCR), an orphan PVC becomes its **own child volume node** (own MCR/MCP + `dataRef` + CSI-VS handle); multi-PVC = **child volume nodes**, never a list on one node — **§3.9**, [`design/volume-node-dual-capture.md`](../design/volume-node-dual-capture.md).
- **Snapshot manifests-only path (N2):** этапы **N2a** / **N2b** — [`design/implementation-plan.md`](../design/implementation-plan.md) **§2.4.1**; **декомпозиция поставки N2b по PR** — **§2.4.2** (тот же файл). Публичные поля статуса N2a, allowlist, **временный MCR** (**ownerRef** на root `Snapshot` для GC in-flight; удаляется контроллером после `MCR Ready=True`, где `Ready=True` означает MCP ready + ownerRef handoff на `SnapshotContent`), **каноническая ссылка на MCP** — `SnapshotContent.status.manifestCheckpointName`, delete root / in-flight capture, download API/ошибки, агрегация N2b, OK vs ownerRef — [`design/snapshot-controller.md`](../design/snapshot-controller.md) **§4.3–§4.7**, **§5.2**, **§8.7**, **§10–§11**. Data-layer и полный export/restore — за пределами N2. При стабильном контракте в API — дополнять этот spec, не дублируя design.
- **N2b — форма графа в статусе (PR1+):** опциональные поля **`status.childrenSnapshotRefs`** на **`Snapshot`** и domain snapshots (элементы JSON: обязательные **`apiVersion`**, **`kind`**, **`name`**) и **`status.childrenSnapshotContentRefs`** на **`SnapshotContent`** (элемент: **`name`**). В Go — **`SnapshotChildRef`** / **`SnapshotContentChildRef`**. Generic **не** резолвит дочерний snapshot «по имени» через перебор registry: **`apiVersion`/`kind`** задают объект однозначно. Parent/back-reference задаётся Kubernetes **`ownerReference`** child snapshot → parent snapshot; это lifecycle/requeue helper, **не** источник состава дерева. Семантика заполнения и orchestration — **§2.4.2** плана и design **§11**. **Инварианты дерева snapshot-run, merge и dedup** — **§3**; дизайн-пакет — [`design/demo-domain-csd/README.md`](../design/demo-domain-csd/README.md), [`design/demo-domain-csd/08-universal-snapshot-tree-model.md`](../design/demo-domain-csd/08-universal-snapshot-tree-model.md).

### §3.0. Две стадии: capture-time domain expansion и обход сохранённого графа

Контракт **§3** разделяет **кто** определяет состав дерева и **как** по нему ходит общий код.

**1) Capture-time domain expansion (построение дерева снимка).** При создании пользователем **`XxxxSnapshot`** **доменный** контроллер соответствующего типа определяет **snapshot scope** для этого ресурса и то, **как** каждый охваченный объект представлен в дереве run, в частности:
- обычные Kubernetes-ресурсы → **manifest capture** данного узла (MCP / MCR и пр. — по нормативам N2a/N2b и согласованным design);
- ресурсы **volume / data** → привязка **PVC manifest + VSC** в **одном ownership scope** на соответствующем `SnapshotContent` (singular **`status.dataRef`** + MCP targets на том же узле); root **`Snapshot`** — **aggregator** (`dataRef=nil`, residual MCP; дети = реальная domain decomposition **+ дочерние volume-узлы** для orphan PVC) — **§3.9**, design [`volume-node-dual-capture.md`](../design/volume-node-dual-capture.md);
- объекты, для которых предусмотрен **другой** доменный snapshot (другой **`YyyySnapshot`** / контроллер) → оформляются как **явные дочерние** **`YyyySnapshot`** (и связанный **`SnapshotContent`**), с публикацией рёбер в **`childrenSnapshotRefs`** / **`childrenSnapshotContentRefs`** по политике родительского узла.

**Логическое дерево** snapshot-run материализуется **сверху вниз** на этом этапе. **Generic**-код **не** выводит состав дерева из инвентаря API и **не** подменяет решение домена; он **использует только уже записанные** в **`status`** refs (**§3.1**, **INV-REF1**).

**2) Traversal уже построенного snapshot-графа (read-path / generic).** После публикации refs **generic** и сценарии **aggregate / download** обходят узлы **только** по разрешённым цепочкам **`childrenSnapshotRefs`** и **`childrenSnapshotContentRefs`** (этот spec или согласованный под-документ). На этой стадии **MUST NOT:** заново **обнаруживать** (discovery), какие объекты кластера «входят в снимок», и **MUST NOT:** **восстанавливать** или **достраивать** дерево **list-перечислением**, фильтрами по namespace или иными эвристиками **вместо** или **в обход** недостающих или неполных refs.

**INV-REF-C1**, **§3.4** и **§3.5** относятся к стадии **(2)** и к обязанности считать границу run **замкнутой** по сохранённому графу.

### §3.1. Логическое дерево и источник истины

- **MUST:** логическое дерево snapshot-run задаётся **только** refs-полями **`status`** на **`XxxxSnapshot`** / **`SnapshotContent`**, **опубликованными** на пути от **root** **`Snapshot`** этого run (материализация дерева — **§3.0** п. 1; обход без расширения scope — **§3.0** п. 2): **`childrenSnapshotRefs`** — **основной** носитель **ребёнка-узла** в дереве; **`childrenSnapshotContentRefs`** — **дополняющий** слой **только** там, где это **нормативно** требует этот spec или согласованный под-документ (read-path по графу, aggregation, политика этапа), **без** подмены SoT, задаваемого **snapshot** refs (см. [`05`](../design/demo-domain-csd/05-tree-and-graph-invariants.md) §2, абзац **Snapshot refs vs content refs**).
- **Область:** **`XxxxSnapshot`** / **`SnapshotContent`** **могут** существовать в API и reconciler'иться **доменным** (или иным) контроллером **до** появления в **`children*Refs`** данного run или **вне** любого такого run — это **не** запрещает **§3** и не отменяет **CSD** / reconcile зарегистрированных типов (**§0**). **§3** задаёт **только** состав **логического дерева** конкретного run от root **`Snapshot`** и обязанности **generic** (обход, dedup/exclude и т.д.) относительно **этого** дерева.
- **MUST NOT:** объект считаться узлом этого дерева, если он **не** представлен в **`children*Refs`** на пути от root (даже если существует в API). (**INV-REF1**, см. [`05`](../design/demo-domain-csd/05-tree-and-graph-invariants.md) §1.)

### §3.2. Parent-owned refs и ключ элементов refs

- **MUST:** логическое дерево одного snapshot-run **namespace-local** относительно **root** **`Snapshot`**: все узлы дерева (**`Snapshot`**, дочерние **`XxxxSnapshot`** из **`childrenSnapshotRefs`**, связанные namespace-scoped объекты этого run в смысле контракта модуля) **MUST** находиться в **одном** namespace — том же, что и root **`Snapshot`**. **MUST NOT:** поддерживать или подразумевать **cross-namespace** рёбра в **`childrenSnapshotRefs`**. Namespace дочернего snapshot не задается в ref и всегда неявно равен namespace родителя. Поиск parent **`Snapshot`** по событию дочернего snapshot **MUST** выполняться **только** в namespace дочернего объекта (**без** cluster-wide list ради «родителя в другом namespace»).
- **MUST:** каждый parent snapshot controller **владеет** своими исходящими рёбрами и записывает полный **`status.childrenSnapshotRefs`** для своего snapshot-узла. Для **`Snapshot`** этот список строится через CSD/registry discovery top-level domain resources; для других domain controllers — из собственной domain model. Child controllers **MUST NOT** self-register: они не патчат **`parent.status.childrenSnapshotRefs`** и не патчат parent **`childrenSnapshotContentRefs`**.
- **MUST:** root **`Snapshot`** treats CSD `priority` as layers/waves, not just item ordering. Lower numeric priority is planned first. A whole priority layer is processed together: all mappings in that layer are listed, all required child snapshots are created/ensured, and then traversal stops until every planned/existing snapshot in that layer has `PlanningReady=True` (gen-gated). Higher-numbered-priority mappings **MUST NOT** be planned before coverage from the completed lower-numbered (earlier) layer is visible through the snapshot tree. If a child snapshot in the current layer reaches terminal failure (`PlanningReady=False/GraphPlanningFailed` or terminal `Ready=False`), root **MUST** set an explicit `PlanningReady=False` condition/error and stop traversal of later (higher-numbered) layers instead of waiting forever. Retry remains possible after the child state is fixed.
- **MUST (framework snapshot contract):** generic orchestration **MUST NOT** предполагать domain-specific spec layout. Контракт участника дерева делится на три категории:
  - **Generic-owned metadata (source identity):** каждый CSD-registered **`XxxxSnapshot`** в дереве (включая дочерние, создаваемые domain controllers внутри subtree; кроме root **`Snapshot`**) **MUST** нести зарезервированную аннотацию **`metadata.annotations["state-snapshotter.deckhouse.io/source-ref"]`** — JSON `{apiVersion, kind, namespace, name, uid}` на namespace-local source object, который этот snapshot materializes. Это **единственный** источник истины source identity для generic coverage; **`uid`** обязателен (пересоздание source с тем же именем не считается покрытым старым snapshot). Контроллеры **MUST** сохранять эту аннотацию; planner fail-closed при её отсутствии/дрейфе (`PlanningReady=False/SourceIdentityAnnotationMismatch`), **без** silent rewrite.
  - **Framework status protocol (фиксированные канонические имена):** generic вправе читать **`status.boundSnapshotContentName`**, **`status.childrenSnapshotRefs`**, **`status.conditions[PlanningReady|Ready]`**. CSD-registration **MUST** fail-closed (`Accepted=False/SnapshotContractUnsatisfied`), если snapshot-CRD структурно объявляет схему, но в ней **отсутствуют** канонические status-поля. На регистрации энфорсятся **только** **`status.boundSnapshotContentName`** и **`status.conditions`** (shallow проверка наличия путей, не полная семантика). Проверка **lenient** (не блокирует), когда схема структурно не интроспектируема: версия без `openAPIV3Schema`, root c `x-kubernetes-preserve-unknown-fields`/без properties, или `status`-подсхема preserve-unknown. **`status.childrenSnapshotRefs`** — каноничное поле протокола, обязательно для **non-leaf** snapshot kinds, но **не энфорсится** на регистрации (leaf-ness неизвестна на этапе CSD и leaf kinds правомерно его опускают).
  - **Domain spec — opaque:** generic **MUST NOT** читать доменный spec и **MUST NOT** требовать `spec.sourceRef` или иное доменное поле. Доменный контроллер **MAY** выводить своё `spec.sourceRef` (или аналог) из аннотации (one-shot, затем immutable) — это demo/manual API-compat, не часть generic-контракта.
  - Положение snapshot в дереве публикует parent через **`status.childrenSnapshotRefs`**; Kubernetes **`ownerReference`** child → parent используется только для lifecycle/back-reference/requeue и **не** задаёт состав дерева. **`Snapshot`** (root) не имеет source identity аннотации: его source неявно равен **`metadata.namespace`**. Source mappings в этом slice — **namespaced-only** (cluster-scoped source вне scope).
- **MUST:** канонический ключ элемента refs: для **`SnapshotChildRef`** — **`(apiVersion, kind, name)`**; для **`SnapshotContentChildRef`** — **`name`** дочернего **`SnapshotContent`**. Ключ **MUST** совпадать с OpenAPI CRD этого репозитория.

### §3.2.1. Own scope filtering

- **MUST:** own materialization scope каждого domain controller ограничен ресурсами, принадлежащими текущему domain object, плюс явно разрешёнными root/namespace-level ресурсами. Если resource имеет **`ownerReferences`** и они указывают на другой domain object, controller **MUST NOT** включать такой resource в свой own MCR; он должен быть покрыт child snapshot или fallback namespace-level логикой.
- **MUST:** child graph delegation имеет приоритет над own scope: resource, вынесенный в child snapshot subtree, не должен дублироваться в own MCR родителя.
- **MUST:** каждый materialized content-node имеет собственный **`status.manifestCheckpointName`** на MCP только своего own scope; parent MCP **MUST NOT** служить контейнером всего subtree. MCP **может быть пустым** (0 objects), если own scope после exclude пустой. Объединение parent + child MCP выполняется только на read-path по **`childrenSnapshotContentRefs`**.
- **MUST:** own scope **`Snapshot`** состоит только из namespace-scoped resources из root allowlist. **`Snapshot`** остаётся namespaced built-in snapshot type, не регистрируется через CSD, и **не** захватывает cluster-scoped Kubernetes **`Namespace`** object. Resolved target namespace = **`Snapshot.metadata.namespace`**; **`spec.namespace`** / **`spec.targetNamespace`** не вводятся.

### §3.2.2. Snapshot lifecycle ownerRef model

```text
Root Snapshot:
  not owned by root ObjectKeeperWithTTL
  root ObjectKeeperWithTTL.spec.followObjectRef -> root Snapshot
Child Snapshot:
  ownerRef -> parent Snapshot

Every SnapshotContent MUST have exactly one lifecycle owner.
Root SnapshotContent:
  ownerRef -> root ObjectKeeperWithTTL
Child SnapshotContent:
  ownerRef -> parent SnapshotContent

Artifacts referenced by SnapshotContent.status:
  ownerRef -> owning SnapshotContent
```

- **MUST:** root `ObjectKeeper` in `FollowObjectWithTTL` mode is the lifecycle anchor for one snapshot-run. Root `SnapshotContent` has ownerRef to the root ObjectKeeper.
- **MUST:** `spec.followObjectRef` points to the root Snapshot.
- **MUST:** the ObjectKeeper MUST NOT have ownerRef to Snapshot. A root Snapshot is the follow target of ObjectKeeperWithTTL, not an owned object. A root Snapshot MUST NOT be owned by ObjectKeeper. When TTL expires and OK is removed, Kubernetes GC can remove the retained `SnapshotContent` tree and artifacts owned by contents.
- **MUST:** a Snapshot without parent Snapshot ownerRef is a root Snapshot. Any Snapshot kind can be root and must be restorable/exportable independently, including standalone demo Snapshot kinds.
- **MUST:** child Snapshot ownerRef points to its parent Snapshot; child `SnapshotContent` ownerRef points to its parent `SnapshotContent`; MCP / future VSC or equivalent final data artifacts ownerRef points to the owning `SnapshotContent`.
- **MUST:** a child Snapshot MUST have ownerRef to parent Snapshot.
- **MUST NOT:** leave child Snapshot or `SnapshotContent` without lifecycle ownerRef after reconcile convergence.
- **MUST NOT:** set `SnapshotContent.ownerReferences` to a short-lived Snapshot.
- **MUST NOT:** silently steal Snapshot or `SnapshotContent` from another parent/root lifecycle owner. Conflicting lifecycle ownerRef is a fail-closed condition; unrelated non-controller ownerRefs may be preserved.
- **MUST:** a same namespace/name root Snapshot run reuses the retained root ObjectKeeper only for the same snapshot UID recorded in `ObjectKeeper.spec.followObjectRef.UID`. If the old run's root ObjectKeeper is still retained and follows an old Snapshot UID, a newly created root Snapshot with the same namespace/name MUST fail closed until that ObjectKeeper expires or is removed; it MUST NOT reuse that retained ObjectKeeper.
- **NOTE:** temporary ObjectKeeper protection for execution artifacts before handoff is separate from the root TTL ObjectKeeper. Generic execution ObjectKeepers for MCR/VCR-style requests are UID-aware so a recreated request with the same namespace/name does not collide with stale protection from an old request UID. Request objects such as MCR/VCR remain temporary and may be owned by the Snapshot for in-flight GC; they are deleted only after final artifacts are bound to `SnapshotContent`.

### §3.3. Удаление элемента из refs

- **MUST:** удаление записи о дочернем узле из **`children*Refs`** родителя выполнять **только** контроллер, который **владеет** соответствующим дочерним объектом (создал и ведёт его), **или** по **явной** политике reconcile родителя, задокументированной в коде модуля и не противоречащей **INV-REF-M2**.

### §3.4. Generic `Snapshot` и read-path по API (только стадия 2)

Обязанности generic здесь — **не** domain expansion (**§3.0** п. 1), а **обход уже опубликованного** графа и сопутствующие правила fail-closed (**§3.0** п. 2).

- **MUST:** для subtree exclude / root capture и read-path по сохранённому графу generic резолвит дочерние **`XxxxSnapshot`** → **`status.boundSnapshotContentName`** и обходит дочерние **`SnapshotContent`** через **`pkg/snapshot.GVKRegistry`** (CSD/bootstrap + **`RESTMapper`**), с **`unstructured`** Get по зарегистрированным GVK. При обходе дочерних `Snapshot` по **`status.childrenSnapshotRefs`** generic **MUST** загружать каждого ребёнка **одним** **`client.Get`** по **`apiVersion`/`kind`** из ref (**строгий Kubernetes-style ref**). Это используется только для (a) барьера волны и (b) bridge терминального capture-failure дочернего `Snapshot`; итоговый `Ready` это **не** вычисляет — snapshot-level `Ready` только зеркалит bound `SnapshotContent.Ready` (§3.8). **MUST NOT:** сканировать registry по всем snapshot-GVK для выбора ребёнка и **MUST NOT:** вводить состояние «ambiguous ref». **MUST NOT:** импортировать пакеты доменных CRD (в т.ч. demo) или хардкодить имена доменных snapshot/content kinds в generic usecase.
- **MUST:** E5 subtree exclude строится как union object identities из MCP всех materialized descendant content-nodes, достижимых через **`childrenSnapshotContentRefs`** common **`SnapshotContent`**. Missing/NotReady descendant MCP — fail-closed pending; child manifests не записываются в parent MCP.
- **MUST:** при каждом успешном refresh графового registry текущее множество пар пересобирается целиком (graph built-ins ∪ eligible CSD, затем фильтр по **`RESTMapper`**); частичное «домешивание» без полной пересборки — вне контракта. Built-in graph registry содержит только **`Snapshot` → `SnapshotContent`**; demo VM/Disk пары появляются в graph registry только через eligible CSD. **MUST NOT:** превращать read-path в polling: при промахе зарегистрированного dedicated-content GVK разрешено **не более одной** попытки **`TryRefresh`** на одну пользовательскую операцию reconcile/API, затем снова одна попытка чтения; при повторном промахе — **fail-closed**. После удаления CSD или потери eligibility соответствующие пары исчезают из registry; уже сохранённый heterogeneous граф с такими узлами может стать **нечитаемым** для generic — **fail-closed**, без list-эвристик (**совместимо с INV-REF-C1 / fail-closed выше**).
- **MUST NOT:** reconciler **`Snapshot`** (и общий код exclude/dedup для root capture) **достраивать** логическое дерево или множество узлов из **list/search по namespace** или эвристик без ref на пути от root. (**INV-REF1**.)
- **MUST NOT:** при отсутствии или пустоте **`childrenSnapshotContentRefs`** там, где поле предусмотрено схемой, **самостоятельно** находить **`*SnapshotContent`** через list API без нормативного правила в этом spec или в согласованном под-документе (например расширение PR4 read-path). **По умолчанию** поведение в такой ситуации — **fail-closed** (не продолжать этап). **Явный fallback** (в т.ч. только из цепочки **snapshot refs**) **допустим только** если он **отдельно** закреплён в этом spec или в согласованном под-документе; иначе list/search «для восстановления content» — **вне** контракта. Конкретный разрешённый вариант **MUST** быть указан в реализации и тестах до включения поведения в релиз. (**INV-REF-C1**.)
- **TODO: retained manifest read API:** current `/snapshots/{name}/manifests` route is semantically a live Snapshot route. After the Snapshot object is deleted, this route should return `404 Snapshot not found`. Retained manifests should be read through a durable content route: `/snapshotcontents/{contentName}/manifests`, because `SnapshotContent` is the retained source of truth after Snapshot deletion. Do not rely on resolving deleted Snapshot name through root ObjectKeeper as the long-term API contract. ObjectKeeper is lifecycle/TTL infrastructure, not the retained read identifier. Current root ObjectKeeper-based retained lookup is temporary behavior / implementation detail until the durable content route exists.

### §3.5. Граница run и fail-closed dedup

- **MUST:** вычисление dedup / exclude для root capture и **сопутствующий** подбор связанных объектов (обнаружение MCP/VS и т.п. **в контексте** покрытия и исключений) выполнять **только** в пределах дерева **текущего** snapshot-run (обход от root **`Snapshot`** по **`children*Refs`**), **без** расширения множества узлов за пределы этого обхода под видом «подготовки данных». (**INV-S0**, [`design/demo-domain-csd/06-coverage-dedup-keys.md`](../design/demo-domain-csd/06-coverage-dedup-keys.md).)
- **MUST NOT:** при невозможности **надёжно** построить множества exclude расширять dedup или исключения «по догадке» по неполным данным; поведение **fail-closed** — как в **INV-E1** ([`06`](../design/demo-domain-csd/06-coverage-dedup-keys.md) §4).

### §3.6. CSD и ownerRef (напоминание)

- **MUST NOT:** использовать **CSD** или **`ownerReference`** как источник истины **состава** логического дерева или для **dedup**; **CSD** — регистрация типов / runtime watches (**§0**); **`ownerReference`** — lifecycle / GC ([`design/demo-domain-csd/08-universal-snapshot-tree-model.md`](../design/demo-domain-csd/08-universal-snapshot-tree-model.md) часть B).

#### §3.6.1. Topology vs coverage (нормативная формула — `INV-TOPO-COV`)

Source-сторона иерархии (например `DemoVirtualMachine.spec.virtualDiskName -> DemoVirtualDisk`, `DemoVirtualDisk.spec.persistentVolumeClaimName -> PersistentVolumeClaim`) и snapshot-сторона покрытия — **разные** вещи и не должны смешиваться:

- **topology source: spec links downward** — иерархию задаёт **только** направленная вниз spec-ссылка родителя на ребёнка по имени в том же namespace (parent → child). Никакой другой сигнал не задаёт топологию.
- **ownerReferences are not topology** — Kubernetes `ownerReferences` на source-объектах **MUST NOT** трактоваться как источник топологии дерева; они — lifecycle / GC.
- **reference does not imply coverage** — наличие spec-ссылки (parent ссылается на child) **само по себе не означает**, что child покрыт. Ссылка — это намерение/топология, не факт захвата.
- **top-level exclusion happens only after actual child snapshot coverage** — ресурс исключается из top-level (не становится прямым ребёнком root) **только после фактического покрытия** реальным потомком: child snapshot опубликован и покрытие наблюдаемо через дерево (`status.childrenSnapshotRefs` / descendant `SnapshotContent.status.dataRef` (incl. child volume nodes) / in-flight VCR targets — **§3.5**). Если родитель не снапшотится (его kind не eligible через CSD), его child **не** покрыт и захватывается standalone как top-level — ровно так же, как nested PVC становится orphan-PVC, если его диск не снапшотится.

### §3.8. Parent `Snapshot`: `Ready` mirrors `SnapshotContent`

Когда **`status.childrenSnapshotRefs`** **не пуст**, `Snapshot` и другие snapshot controllers **MUST NOT** вычислять финальный `Ready` по дочерним snapshot-объектам. Единственный критерий истины: **`Snapshot.status.conditions[Ready]` зеркалит bound `SnapshotContent.status.conditions[Ready]`**. Если content теряет `Ready=True`, snapshot также становится `Ready=False` при следующем reconcile/watch event. Так как `SnapshotContent` не хранит reverse ref на snapshot, controllers **MUST** обеспечивать mirror через собственные watch/requeue механизмы (например polling `RequeueAfter` пока bound content pending).

`childrenSnapshotRefs` остаются snapshot-level planned graph. Parent snapshot controller после `PlanningReady=True` (gen-gated, **INV-COND3**) резолвит детей в bound child `SnapshotContent`, публикует durable **`SnapshotContent.status.childrenSnapshotContentRefs`**, и только после этого публикует result refs вроде **`status.manifestCheckpointName`**. **`SnapshotContentController`** не читает live snapshot, не читает MCR/VCR и не строит граф; он валидирует уже опубликованные **`manifestCheckpointName`**, **`dataRef`** (singular), **`childrenSnapshotContentRefs`**, затем агрегирует parent content `Ready` (**§3.9.7**, единственный агрегатор). **SnapshotGraphRegistry** на этом пути не является источником истины. Пробуждение parent snapshot при изменении дочернего объекта — через динамические watches на зарегистрированные snapshot-GVK (см. **§0**) и map **child → parents** по совпадению ref **в пределах namespace дочернего объекта** (**§3.2**).

- **MUST (INV-FAIL-PROP, parent invalidation — консолидированный инвариант):** parent `Snapshot`/`SnapshotContent` `Ready=True` **тогда и только тогда**, когда **все обязательные** потомки/артефакты **присутствуют и healthy**. Любой обязательный потомок/артефакт, который **удалён**, **отсутствует**, стал `Ready=False`/`Failed`, или потерял свой required data artifact / MCP / chunk / VSC, **MUST** приводить к `Ready=False` у родителя с **детерминированным** reason и **MUST** допускать восстановление `Ready=True` после recreate/recovery. Это применяется **одинаково** к: child `Snapshot`, child `SnapshotContent` (включая дочерние volume-узлы), `ManifestCheckpoint`, `ManifestCheckpointContentChunk`, `VolumeSnapshotContent`, orphan `VolumeSnapshot` handle → дочерний volume-узел (через durable VSC, см. **§3.9.11**) и `dataRef` artifact target. Корректность всегда производится reconcile-ревалидацией от truth refs (**INV-RECONCILE-TRUTH**); watches/indexers — только liveness-оптимизация и **MUST NOT** быть единственным условием инвалидции.
- **MUST (child-snapshot delete wake-up):** create/update/**delete** события дочернего snapshot-объекта **MUST** будить каждого parent `Snapshot`, чей `status.childrenSnapshotRefs[]` ссылается на этот child (map child→parents в пределах namespace ребёнка, **§3.2**). На **delete** объект ребёнка уже отсутствует в API, поэтому relay **MUST** реконструировать его identity из watched GVK + ключа события и всё равно enqueue'ить родителей — иначе удаление обязательного child Snapshot не было бы event-driven (инвалидция полагалась бы только на content-side ownerRef-цепочку и polling fallback). Это не заменяет revalidate-from-truth, а лишь делает инвалидцию/recovery быстрой.
- **MUST (relay registration-race requeue, no hot-loop):** короткий `RequeueAfter`-ретрай в child-snapshot relay для «bound child, ещё не перечислен родителем» **MUST** применяться **только** к прямым детям unified `Snapshot` (т.е. объект имеет controller-`ownerReference` на `Snapshot` группы `storage.deckhouse.io`). Головной/root `Snapshot` (**нет** ownerReference) и внуки, принадлежащие доменному snapshot (например disk-snapshot под VM-snapshot, propagation которых идёт по content-ownerRef-цепочке), **MUST NOT** попадать в этот ретрай — иначе relay бесконечно реквичит объект, который ни один unified Snapshot не перечислит (у `RequeueAfter` нет cap по попыткам), занимая reconcile-воркер.
- **MUST (INV-FAIL1, ancestor-chain propagation):** терминальный failure листового `SnapshotContent` (нет/`Ready=False` MCP, отсутствующий/сломанный data artifact, унаследованный `ChildrenFailed`) поднимается **только** по ancestor-chain к root: каждый предок получает `ChildrenReady=False` / `Ready=False` с reason `ChildrenFailed`, а `message` сохраняет имя failed-потомка и исходный reason/message листа (кумулятивно по глубине). Sibling-ветки **MUST NOT** менять состояние. Терминальные content-reason'ы: `ManifestCheckpointFailed`, `DataArtifactInvalid`, `DataArtifactNotSupported`, `ArtifactMissing`, `ChildrenFailed`; нетерминальные (`ArtifactNotReady` и т.п.) поднимаются как `ChildrenPending`.
- **MUST (INV-COND4, mirror):** `Snapshot.Ready := mirror(SnapshotContent.Ready)` (status/reason/message копируются verbatim). До bind (нет bound content или у content ещё нет `Ready`) допустим локальный transitional `Ready=False/ContentBindingPending`. После bind контроллер **MUST NOT** писать локальный semantic `Ready` state на `Snapshot` — **ни pending, ни terminal** — только mirror (eventual consistency: если content ещё не пересчитался, mirror даёт `ContentBindingPending` и reconcile requeue'ится). В частности, terminal MCP failure после bind **MUST** идти через content (`ManifestsReady=False/ManifestCheckpointFailed`) и зеркалиться, а **не** писаться локально. Единственное исключение — bridge для failures, которые дерево контента ещё не может выразить: терминальный child-Snapshot capture failure до появления bound content и pre-publish capture-planning failures (build/list targets, plan drift), возникающие до публикации requests в content.

- **MUST:** при **NotFound** дочернего snapshot или bound content — parent `SnapshotContent Ready=False` как ожидание (ordering / ещё не создан). При ошибке Get (не NotFound) — fail операции агрегации на этом reconcile. Состояния «ambiguous ref» **нет** в контракте.
- **MUST:** классифицировать каждого child content как **успех** (`Ready=True`), **ожидание**, или **терминальный провал** (`Ready=False` с причиной из согласованного набора terminal capture / fail-closed). Расширенные мотивы — [`design/demo-domain-csd/07-ready-delete-matrix.md`](../design/demo-domain-csd/07-ready-delete-matrix.md).
- **MUST:** при одновременном наличии нескольких причин на parent content выбирать **одну** итоговую причину **`Ready=False`** в строгом приоритете. Snapshot-level `Ready` не имеет собственной матрицы: он копирует condition с bound `SnapshotContent`.

### §3.9. Logical SnapshotContent: manifest + singular `dataRef` (N5+, Variant A)

Нормативное резюме **целевой** модели (**Variant A**, cardinality ≤1). Детали, фильтры, PR roadmap — [`design/volume-node-dual-capture.md`](../design/volume-node-dual-capture.md), [`implementation-plan.md`](../design/implementation-plan.md) **§2.4.5**. **`SnapshotContent.status.dataRef`** — singular object (не массив), enforced структурно типом CRD (`type: object`; CEL не используется); multi-PVC выражается **только** дочерними volume-узлами (`childrenSnapshotContentRefs[]`).

#### §3.9.1. What a SnapshotContent represents

- **MUST:** **`SnapshotContent`** represents a **logical snapshot node** (domain object, namespace root, decomposed child such as disk, **or child volume node** for an orphan/loose PVC). It carries **at most one** data artifact.
- **MUST:** each logical node exposes:
  - **`status.manifestCheckpointName`** → **MCP** built from **MCR `spec.targets[]`** (zero or many manifest objects);
  - **`status.dataRef`** → **at most one** binding **{ PVC target identity, cluster-scoped data artifact }** (singular object) — **единственный** контракт data artifacts;
  - **`status.childrenSnapshotContentRefs[]`** → zero or more child logical nodes: **real** domain decomposition **and** child volume nodes (one per orphan/loose PVC).
- **MUST NOT:** use a list **`status.dataRefs[]`** (removed; structurally a node cannot hold ≥2 bindings); **no** fallback, **no** dual-write. The generic `DataRefList()` bridge exposes the singular `dataRef` as a 0/1 slice for slice-based helpers.

#### §3.9.2. Co-ownership: PVC manifest and data binding

- **MUST:** for every PVC whose data is captured, the **sanitized PVC manifest** and the matching **`dataRef`** (same **`pvcUID`**) **MUST** reside on the **same** `SnapshotContent`. Since a node carries ≤1 `dataRef`, that content is the **node owning that PVC** — a **child volume node** for an orphan/loose PVC (its own MCP holds only that PVC manifest), or the domain leaf for a domain-owned PVC.
- **MUST NOT:** duplicate the same PVC manifest or data binding across ownership scopes in one run; the root aggregator **MUST NOT** place a residual/orphan PVC manifest on the root MCP **at all** — residual root-owned PVC manifests **MUST** be excluded from the root MCR **up front** (unconditionally, independent of whether the child volume node already exists), because the orphan `VolumeSnapshot` binding is asynchronous and would otherwise race the (near-instant) root manifest leg into double-capturing the manifest on both the root and the child.
- **MUST:** final PVC **ownership** for a run **MUST** be resolved by **snapshot-run tree coverage**, **domain claims**, **policy exclusions**, and **`pvcUID` dedup** (**INV-S0**, **INV-P1**). **MUST NOT** define ownership **solely** by listing all PVCs in a namespace.

#### §3.9.3. Namespace root vs non-root domain

- **MUST:** root namespace **`Snapshot`** / bound root **`SnapshotContent`** is an **aggregator**: MCP holds **residual** namespace manifests only (allowlist minus E5, possibly **empty archive**); **`status.dataRef` MUST be nil**; **`children*Refs`** represent **real** domain decomposition **and** one **child volume node** per residual/orphan PVC.
- **MAY:** for **residual root-owned PVCs only**, use a **namespace-wide PVC list** as **candidate discovery**: `PVC candidates − subtree-covered pvcUIDs − policy exclusions = residual root-owned PVC`. Listing is a **technical** step; **ownership** of each candidate **MUST** still follow tree coverage and dedup before a child volume node is created for it.
- **MUST:** root **MUST NOT** create a `dataRef` (or a child volume node) for PVCs already covered by a child/domain subtree in this run.
- **MUST:** non-root / domain **`SnapshotContent` MUST NOT** be **data-only**: MCP **MUST** include at least the manifest of the snapshot's **source object** (identified by the generic source identity annotation, §3.2); if the node owns a PVC, MCP **MUST** also include that PVC's sanitized manifest; **`dataRef`** holds the matching VSC binding.
- **MUST:** if the **source object** (per the source identity annotation) is **NotFound** or deleted between planning and manifest capture for a non-root domain node, reconcile **MUST** fail-closed (terminal failure or explicit pending reason). **MUST NOT** reach capture-complete / data-only success without the source object manifest in MCP.

#### §3.9.4. Children vs multi-PVC (Variant A)

- **MUST:** **multi-PVC** on one logical scope **MUST** be represented as **child volume nodes** — one `SnapshotContent` per PVC, each with its own MCP (that PVC's manifest) and its own singular **`dataRef`** — linked via **`childrenSnapshotContentRefs[]`**. A node **MUST NOT** carry more than one `dataRef` (the field is a singular object; ≥2 is unrepresentable).
- **MUST:** **`children*Refs`** carry both **real** domain decomposition (separate snapshot kinds / lifecycle) **and** child volume nodes for orphan/loose PVCs.
- **MUST:** the target architecture **is** “one volume → one `SnapshotContent` → one `dataRef`” for every captured volume; a domain leaf owns exactly one PVC, and an orphan PVC gets its own child volume node (see **§3.9.11**).

#### §3.9.5. Requests (MCR / VCR / orphan VS)

- **MUST:** **`ManifestCaptureRequest`** is a manifest capture request: **`spec.targets[]`** lists manifest objects; **`status.checkpointName`** references the durable **MCP**. A **child volume node** has its **own** per-orphan MCR whose targets are **only** that PVC.
- **MUST:** a **domain** **`VolumeCaptureRequest`** scopes **`spec.targets[]`** to the **one** PVC the leaf owns; **`status.dataRefs[]`** (foundation contract, still a list) is consumed as a single binding. If a domain content's VCR reports **>1** binding, reconcile **MUST** fail-closed terminal (`VolumeCaptureFailed`), never silently take `[0]`.
- **MUST:** at most **one active MCR** and (for a domain leaf) **one active VCR** per logical **`SnapshotContent`**.
- **MUST:** the **root MUST NOT** create a VCR for residual/orphan PVCs; those use the orphan VS leg (**§3.9.11**).

**storage-foundation:** the VCR contract (`spec.targets[]`, `status.dataRefs[]`, aggregate **Ready**) and the extended-VS fork (`status.boundSnapshotContentName`) are foundation-owned — see [`implementation-plan.md`](../design/implementation-plan.md) **§2.4.5** **PR-F**.

#### §3.9.6. Capture execution and publish

- **MUST:** a **domain leaf** runs one capture plan: **one MCR** (`spec.targets[]` = manifest scope) and **in parallel one VCR** (`spec.targets[]` = the one PVC). A **child volume node** runs its own per-orphan MCR + binds a CSI `VolumeSnapshot` → VSC. Execution requests **MUST** use **ownerRef → owning Snapshot**; durable artifacts **MUST** use **ownerRef → owning SnapshotContent** after handoff.
- **MUST:** publish on the **owning** `SnapshotContent`: **`status.manifestCheckpointName`** ← **`MCR.status.checkpointName`**; **`status.dataRef`** ← the single binding (domain `VCR.status.dataRefs[0]`, or orphan VS-bound VSC). Published refs **MUST NOT** reference **VolumeCaptureRequest**, **ManifestCaptureRequest**, `VolumeSnapshot`, or other execution requests — the durable artifact is always a `VolumeSnapshotContent`. The **root** `dataRef` stays **nil**; orphan bindings are published on the **child volume node**, not the root.
- **MUST:** snapshot temporary status uses **`manifestCaptureRequestName`** and **`volumeCaptureRequestName`** (singular each); a child volume node tracks its own **`manifestCaptureRequestName`**.

#### §3.9.7. Ready (condition model: `ManifestsReady` / `VolumeReady` / `ChildrenReady` / `Ready`)

Публичная модель — **5 condition'ов**: `PlanningReady` (gate/барьер, §3.8), `ManifestsReady`, `VolumeReady`, `ChildrenReady`, `Ready`. `SnapshotContent` несёт `ManifestsReady` / `VolumeReady` / `ChildrenReady` / `Ready`; `Snapshot` несёт `PlanningReady` (своё) и зеркалит `Ready` (§3.8). Манифестная линия (`ManifestsReady`, MCP) и волюмная линия (`VolumeReady`, data) разнесены ради ops-диагностики: видно сразу по типу condition, чем заблокирована готовность. `PlanningReady` (live-gate, бывш. `ChildrenSnapshotReady`, ещё раньше `DomainReady`) **MUST NOT** появляться на `SnapshotContent` и **MUST NOT** входить в формулу `Ready`.

- **MUST:** агрегат вычисляется ровно на `SnapshotContent` (**INV-COND2**, единственный агрегатор):

```text
SnapshotContent.ManifestsReady = MCP Ready
SnapshotContent.VolumeReady   = (dataRef == nil) ∨ (dataRef.artifact Ready)   // ≤1 binding
SnapshotContent.ChildrenReady  = (∀ required child SnapshotContent Ready=True)   // leaf без детей → True
SnapshotContent.Ready          = ManifestsReady ∧ VolumeReady ∧ ChildrenReady
```

- **MUST:** **`ManifestsReady`/`VolumeReady`** покрывают «свои requests узла»: ≤1 MCR (manifest) + ≤1 binding (volume) (**INV-COND5**). **`MCP Ready`** means **`status.manifestCheckpointName`** references an existing MCP with **`Ready=True`** (including **empty archive**, 0 objects). Applies to **aggregator** root/VM: MCP may be **empty by content**, но **`manifestCheckpointName` MUST be present** и указывать на **Ready** MCP. **`SnapshotContentController` MUST** validate **`manifestCheckpointName`** одинаково для любого узла. **MUST NOT** вводить специальную ветку «root без MCP» в v1.
- **MUST:** manifest-only / aggregator nodes treat a **nil** `dataRef` as satisfying the data leg (**INV-COND6**, `VolumeReady=True`); a node that owns a PVC **MUST** remain `VolumeReady=False` until its single binding is Ready (`DataCapturePending` surfaces as "0/1 ready"). Multi-volume readiness is aggregated via **ChildrenReady** over the child volume nodes.
- **MUST (sequencing, v1):** волюмная линия оценивается **только после** готовности манифестной. Пока `ManifestsReady != True`, `VolumeReady=Unknown` с reason `ManifestCapturePending` (линия ещё не оценивалась — это не отказ тома). Независимая оценка линий — future follow-up. Validator order остаётся manifest → data (all refs) → children.
- **MUST (приоритет reason у `Ready`):** при нескольких упавших ногах несётся один reason в строгом порядке `manifestsFailed > volumesFailed > childrenFailed > manifestsPending > volumesPending > childrenPending > Completed` (терминальные провалы первыми; свой узел перед детьми при равной тяжести).
- **MUST (INV-COND1, gate-импликация):** `ManifestsReady=True ⇒ PlanningReady=True` и `VolumeReady=True ⇒ PlanningReady=True`; requests не появляются без планирования. Это держит `Snapshot.Ready == Content.Ready` при том, что `PlanningReady` **не** входит в формулу `Ready`.

#### §3.9.8. Restore and dedup (data path)

- **MUST:** restore-with-data **MUST** match the PVC manifest in a content's MCP to the **`dataRef`** on the **same** content: primary key **`target.uid` (`pvcUID`)**, fallback **`(apiVersion, kind, namespace, name)`** from manifest metadata; missing binding → **fail-closed**. An orphan PVC's manifest + `dataRef` live on its **child volume node**; the resolver reaches it via the CSI `VolumeSnapshot` handle `status.boundSnapshotContentName` and emits the child as its own restore node.
- **MUST:** at most **one** data capture per **`pvcUID`** per snapshot-run (**INV-P1**); manifest E5 exclude remains separate (**INV-S0** / **INV-E1** — [`design/demo-domain-csd/06-coverage-dedup-keys.md`](../design/demo-domain-csd/06-coverage-dedup-keys.md) §4).

#### §3.9.9. Boundaries

- **MUST NOT:** treat **Pod** (or other workload) using a PVC as the **owner** of volume capture; Pod usage may affect **`status.dataConsistency`** (or equivalent) on the owning snapshot node only.
- **MUST NOT:** implement VCR/VSC CSI driver wiring, Ceph/Rook sidecar compatibility, or shadow **`VolumeSnapshot`** workarounds in **state-snapshotter**; that remains **storage-foundation**. **Exception (narrow):** root residual capture of **orphan PVCs** MAY create a **standard** `VolumeSnapshot` (no driver wiring/sidecars) — see **§3.9.11**.

**PVC list vs ownership (residual root):**

- **MAY:** use a **namespace-wide PVC list** **only** as **candidate discovery** for **residual root scope** (see **§3.9.3**): candidates minus subtree-covered **`pvcUID`s**, minus policy exclusions. When computing subtree coverage, **child volume contents are skipped** so their own PVC stays in scope for their leg while their manifest is excluded from the root MCR.
- **MUST NOT:** define PVC **ownership** **solely** by listing all PVCs in a namespace.
- **MUST:** resolve **final ownership** by **snapshot-run tree coverage**, **domain claims**, **policy exclusions**, and **`pvcUID` dedup** before creating a **child volume node** (with its `dataRef` + per-orphan MCR) or adding a PVC manifest to a domain MCR.
- **MUST:** root **MUST NOT** capture (manifest or data) a PVC already covered by a **child/domain subtree** in this run.

#### §3.9.10. Progress and degradation visibility

Design: [`design/status-propagation-and-visibility.md`](../design/status-propagation-and-visibility.md). One behavioural model, two phases (Phase 1 — progress-aware `Ready=False`; Phase 2a — MCP/VSC damaged-artifact wake-up/revalidation, §3.9.10.1). Pending and degradation share the **same** aggregation pipeline and reason taxonomy.

- **MUST (progress-aware):** `Ready=False` **MUST NOT** be mute/generic. `ManifestsReady=False` / `VolumeReady=False` / `ChildrenReady=False` reason+message **MUST** explain what is awaited:
  - manifest leg (`ManifestsReady`): `ManifestCapturePending` (no `manifestCheckpointName` yet / MCP not Ready), `ManifestCheckpointFailed` (terminal);
  - data leg (`VolumeReady`): `DataCapturePending` (published data artifacts not yet ready) with progress count `"<ready>/<total> ready"` and a capped pending list; `ArtifactMissing` / `DataArtifactInvalid` / `DataArtifactNotSupported` (terminal); `ManifestCapturePending` with status `Unknown` while the manifest leg is not yet ready (data leg not evaluated yet);
  - children leg: `ChildrenPending` with progress count `"<ready>/<total> ready"` and a capped pending list; `ChildrenFailed` (terminal).
- **MUST:** the empty-`manifestCheckpointName` message is generic (`waiting for manifest capture checkpoint to be published`). `SnapshotContentController` **MUST NOT** read the live `Snapshot` to enrich a content message (no content→snapshot dependency).
- **MUST (revalidation without watch):** if `SnapshotContentController` is woken while a published artifact ref no longer resolves (e.g. SnapshotContent was `Ready=True`, a later reconcile observes the referenced artifact missing/terminally failed), it **MUST** recompute the affected leg to False (data artifact → `VolumeReady=False`/`ArtifactMissing`; manifest checkpoint → `ManifestsReady=False`/`ManifestCheckpointFailed`) and therefore `Ready=False`, with the artifact kind/name in the message. The **wake-up** mechanism (artifact→content watches) is specified in **§3.9.10.1** (Phase 2a); the recompute itself is part of this model and is unconditional on every reconcile.
- **MUST (leaf-chain diagnostics, content-only):** the terminal `ChildrenFailed` message **MUST** carry, in a canonical parseable form, the direct child, the original failed leaf, and the leaf's original reason/message:

```text
child SnapshotContent <direct-child> failed: leaf=<failed-leaf> reason=<original-reason> message=<original-message>
```

When a parent inspects a terminal child whose reason is `ChildrenFailed`, it **MUST** reuse `leaf`/`reason`/`message` from that child's message and set `direct-child` to that child's name; otherwise the child itself is the failed leaf. This diagnostic is built **only** in `SnapshotContent` aggregation. The namespaced `Snapshot` **MUST NOT** re-traverse children, read child conditions, or build its own diagnostics — `Snapshot.Ready` is a verbatim mirror (the only exception is the terminal child-`Snapshot` capture-failure bridge, §3.8).

##### §3.9.10.1. Phase 2a — durable-artifact degradation wake-up

Phase 2a guarantees that a degraded **durable artifact** (`ManifestCheckpoint`, `VolumeSnapshotContent`) wakes the owning `SnapshotContent`; it adds **no** new condition model. The truth/ownerRef split below is model-wide.

- **MUST (truth vs ownerRef — `INV-TRUTH`):** membership and validation are based **only** on explicit refs (`status.manifestCheckpointName`, `status.dataRef`, `status.childrenSnapshotContentRefs[]`; `MCP.status.chunks[]` is truth for chunks but **out of scope** for status checks in this phase). `ownerReferences` **MUST NOT** decide whether an artifact belongs to a snapshot. `SnapshotContent.Ready = ManifestsReady && VolumeReady && ChildrenReady`; MCP state is surfaced **only** through `ManifestsReady` and VSC/data state **only** through `VolumeReady`; `ChildrenReady` is **not** mutated directly by artifact checks.
- **MUST (no watermark):** classification **MUST** be derived **only** from the current artifact state, never from a prior `Ready=True`. Manifest leg: empty / NotFound → `ManifestCapturePending`; `Ready=False`+terminal `Failed` → `ManifestCheckpointFailed`; `Ready=False` non-terminal / no Ready → `ManifestCapturePending`; `Ready=True` → validate chunks (below), then continue. Data leg: VSC NotFound → `ArtifactMissing` (terminal, since `dataRef` is published only after the VSC exists and is owned); VSC with `deletionTimestamp` set → `ArtifactMissing` (terminal — a durable artifact that is being deleted is not healthy even while it still exists and reports `readyToUse=true`; same terminal classification as full NotFound, so the transition is monotonic ready→missing without a spurious pending step); VSC `readyToUse=false`/no ready → `DataCapturePending` (pending). `ArtifactFailed` **MUST NOT** be inferred from `readyToUse=false` (deferred).
- **MUST (chunk existence by exact ref, get-only):** when the MCP is `Ready=True`, `SnapshotContentController` **MUST** verify that every chunk named in `MCP.status.chunks[]` (truth) still **exists** by an exact GET of `ManifestCheckpointContentChunk/<name>`, stopping at the first `NotFound` → `ManifestsReady=False`/`ManifestCheckpointFailed` (terminal), message `ManifestCheckpoint <mcp> references missing chunk <chunk>`. This is an **existence** check only: it **MUST NOT** read/decode `.spec.data` or verify checksums on reconcile (chunk **content** validation belongs to the explicit read/download/archive path, §3.8 not on every reconcile); the GET **SHOULD** be metadata-only (`PartialObjectMetadata`) so the payload is not transferred. The GET **MUST** use the uncached reader (no cached client Get that would implicitly start a chunk informer); chunks keep **get-only** RBAC (no `list`/`watch`). A transient (non-`NotFound`) error **MUST** requeue, not become terminal. **Chunk deletion does NOT wake reconcile** in this phase (no chunk watch): correctness is produced on the next reconcile (any MCP/VSC/child event), and the read/download/archive path fails immediately on use; a stale `Ready=True` between events is accepted.
- **MUST (wake-up routing, ownerRef-only — `INV-OWNCHAIN`):** artifact create/update/delete events **MUST** be routed to the owning `SnapshotContent` by `ownerReferences` only: `ManifestCheckpoint.ownerRef → SnapshotContent`, `VolumeSnapshotContent.ownerRef → SnapshotContent`. Watch handlers **MUST** only enqueue a `reconcile.Request` and **MUST NOT** write conditions or patch status. A missing/broken ownerRef **MUST** be logged and dropped (no owner guessing). **MUST NOT:** reverse-index wake-up by `status.dataRef`; **MUST NOT:** a `chunk → SnapshotContent` shortcut — chunk routing, if ever added, **MUST** hop only immediate ownerRefs (`chunk → ManifestCheckpoint → SnapshotContent`), never skipping an intermediate parent.
- **MUST (`INV-RECONCILE-TRUTH`):** correctness is produced by `SnapshotContentController` reconcile, which **MUST** recompute state from truth refs on every reconcile; watches are only a liveness/latency optimization. A missed event **MUST NOT** change the eventual state. Each artifact watch **MAY** be registered guarded by RESTMapping so a not-yet-installed CRD degrades to "no watch" rather than failing startup.
- **MUST (ownerRef + Retain self-healing):** on reconcile the owning `SnapshotContent` **MUST** idempotently re-assert its ownerRef on the `VolumeSnapshotContent` referenced by `status.dataRef` (same owner-ref shape as the snapshot-side publisher) **and MUST force `spec.deletionPolicy=Retain`** on it, so ownerRef-only wake-up stays robust without using `dataRef` for routing and the durable artifact cannot silently revert to a non-durable (`Delete`) policy. Data readiness does **not** inspect `deletionPolicy`, so a `Delete`-policy VSC would otherwise pass as healthy yet be non-durable — the self-heal/handoff Retain enforcement is the only guard. A missing VSC is left to data readiness (`ArtifactMissing`); a deleting VSC (`deletionTimestamp` set) **MUST NOT** be patched; this helper **MUST NOT** write status or fail reconcile.
- **Chunk wake-up / deeper validation out of scope (this phase):** `ManifestCheckpointContentChunk` `list/watch` is **not** granted, so chunk deletion does not self-wake (covered above); chunk **content/consistency** beyond existence (e.g. checksum re-verify in conditions) stays on the read/download/archive path. A future `chunk → MCP → SnapshotContent` wake-up (and any chunk `list/watch`) requires an explicit ADR/RBAC decision. Chunk lifecycle is create + ownerRef-cascaded GC (`Controller=true`); the controller does not delete chunks.

#### §3.9.11. Orphan PVC: standalone child volume node (Variant A, INV-ORPHAN4 inverted)

Capture-side handling for **orphan/uncovered PVCs** (namespace PVC − subtree-covered `pvcUID`s − policy exclusions). Under Variant A an orphan PVC is **not** a root data binding nor a visibility-only leaf: it is a **standalone child volume node** in the content tree. Normative rationale: ADR [`snapshot-rework/2026-06-09-orphan-pvc-csi-volumesnapshot.md`](../../../snapshot-rework/2026-06-09-orphan-pvc-csi-volumesnapshot.md).

- **MUST:** the namespace/root controller **MUST NOT** create a `VolumeCaptureRequest` for residual/orphan PVCs. **VCR remains domain-owned** (§3.9.5–§3.9.6 unchanged for non-root/domain nodes).
- **MUST:** for each orphan PVC the root controller **MUST** create a **child volume node** — a cluster-scoped `SnapshotContent` (deterministic name `<rootContentName>-vol-<first12(sha256(<rootContentName>|<pvcUID>))>`, label `state-snapshotter.deckhouse.io/child-volume-node=true`, ownerRef→root `controller=true`, `spec.deletionPolicy` mirrors root) — and link it via `root.status.childrenSnapshotContentRefs[]` (idempotent, optimistic lock).
- **MUST:** the child volume node has its **own** per-orphan `ManifestCaptureRequest` (targets = only that PVC) → its **own** MCP holding that PVC's sanitized manifest; the residual PVC manifest is **excluded from the root MCR up front** (unconditionally, independent of child-node existence — the async orphan VS binding would otherwise race the root manifest leg, §3.9.2). (Co-ownership §3.9.2: PVC manifest + `dataRef` on the **same** content.)
- **MAY:** for the orphan PVC the controller **MAY** create a **standard** `snapshot.storage.k8s.io/v1` `VolumeSnapshot` (`spec.source.persistentVolumeClaimName` = the PVC). No CSI driver wiring / sidecars / vendor compatibility is implemented; the external-snapshotter performs the work.
- **MUST (INV-ORPHAN4 inverted):** after the `VolumeSnapshot` is bound, the shared controller **MUST** write the orphan VS **`status.boundSnapshotContentName`** → the **child volume node** (D4a, merge-patch + `MergeFromWithOptimisticLock` under `RetryOnConflict`). The VS becomes a **namespaced handle to a real content node**, not a visibility-only leaf. Because `status.boundSnapshotContentName` is a Deckhouse **extended-VS fork** field (§3.9.5), the controller **MUST** verify the write **persisted** (re-read the returned object) and **MUST** fail-closed with an actionable diagnostic if it did not — on an upstream VolumeSnapshot CRD the API server silently prunes the unknown field, which would otherwise yield a `Ready`-looking but **unrestorable** snapshot.
- **MUST:** the bound **`VolumeSnapshotContent`** is published into the **child volume node** `status.dataRef` (`target` = source PVC identity incl. `pvcUID`; `artifact` = VSC). The **root** `dataRef` stays **nil**. Data leg readiness/diagnostics reuse §3.9.10.
- **MUST:** the bound VSC is the **durable artifact**: `deletionPolicy=Retain`, ownerRef → the **child volume node** (durable-artifact handoff per §3.9.6). VSC removal is owned by the unified content deletion algorithm.
- **SHOULD (lifecycle / deferred to C5):** if a PVC stops being a residual-root target mid-run (e.g. a domain subtree starts covering it), its already-materialized child volume node (content + per-orphan MCP + retained VSC) is reclaimed via the **root content GC chain** (ownerRef→root `controller=true`) at teardown. Eager **mid-run** pruning of a now-stale child volume node is owned by the unified content deletion / physical-reclaim algorithm (C5) and is intentionally **not** done by the capture path, which **MUST NOT** delete a durable VSC on a transient target-list read (that would risk dropping live data).
- **MUST:** the orphan VS **MUST** remain in `Snapshot.status.childrenSnapshotRefs[]` (tree visibility), now as a **real content child**. The child volume node **MUST** participate in content readiness aggregation (`ChildrenReady`, §3.9.7) and PVC coverage; child volume contents are skipped only when computing *root residual* coverage (§3.9.9) so the orphan PVC stays in scope for its own leg.
- **MUST:** `VolumeSnapshot` objects **MUST NOT** enter any MCP / manifest inventory / aggregated manifests, for **all** VS. The allowlist already excludes VS; an explicit capture-side filter **MUST** keep VS out even if the allowlist is later widened.
- **MUST (restore):** the restore resolver **MUST** reach the orphan PVC's manifest + `dataRef` through the child volume node (via the orphan VS handle `status.boundSnapshotContentName`) and emit it as its own restore node (§3.9.8). The orphan VS is kept alive for the snapshot's lifetime (ownerRef → root `Snapshot`, retained by the root ObjectKeeper TTL), so the resolver currently requires the **live VS handle** to locate the child node and set the PVC `spec.dataSourceRef`.
- **SHOULD (restore robustness, deferred):** the durable artifacts are the retained VSC + the PVC manifest on the child volume node, which are reachable from the **content tree** (`root.status.childrenSnapshotContentRefs` → child volume node) independently of the original VS. A follow-up **SHOULD** let restore fall back to the content-tree edge and **recreate a pre-provisioned VS from the retained VSC** when the original orphan VS no longer exists, removing the hard dependency on the VS surviving. Until then, deleting the orphan VS while the PVC is still uncovered yields a `Ready`-but-unrestorable PVC for that node.
