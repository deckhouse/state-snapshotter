# Design: контракт VolumeCaptureRequest (долгоживущая просьба на один capture)

- **Дата:** 2026-07-22
- **Статус:** Draft (architecture / design) — отложено; проработан контракт до начала реализации
- **Тип:** design/ (модель состояний и инварианты; не нормативный контракт — нормативная выжимка позже уходит в `spec/`)
- **Область:** семантика VCR как **request**, состояния, и правила: когда система имеет право автоматически отказаться от текущего исполнения. Реализация (правки контроллера/SDK, API-поля) — вне документа.

> Документ **не переписывает** план и не содержит задач реализации, кода или TODO. Задача — зафиксировать **контракт VCR**, чтобы вопрос «нужен ли retry / attempt-модель» стал следствием контракта, а не отправной точкой.

---

## 0. Зачем документ и главный вопрос

Обсуждение «как приделать retry к VCR» — преждевременно. Сначала надо ответить на вопрос уровнем выше:

> **Что означает `Failed` у сущности с именем Request — «попробовал и не смог» или «просьба невыполнима»?**

Для request в Kubernetes каноничен второй смысл (PVC не «фейлится» от первой ошибки provisioner'а; `Job` живёт до `backoffLimit`; `CertificateRequest`/`Deployment` многократно реконсилятся). Текущий VCR реализует **первый** смысл: контроллер one-shot («VCR never retries snapshot creation»), и **любая** ошибка в `VolumeSnapshotContent.status.error` делается терминальной (`SnapshotCreationFailed`). Именно смешение «одна попытка мимо» и «невозможно» и порождает потребность в retry как костыле.

Правильный разворот — не «добавить retry», а **сменить контракт VCR** на долгоживущую просьбу. Тогда attempt-модель (`attempt`, `activeRequestRef`, attempt-UID, перенос terminal-решения на «exhaustion») **не нужна** — а в состоянии «неизвестно» она ещё и опасна.

Центральный вопрос, который заменяет «как сделать retry»:

> **В каких состояниях мы вообще имеем право автоматически отказаться от текущего исполнения request?**

---

## 1. Контракт VCR

> **VolumeCaptureRequest — долгоживущая конечная просьба:** «создай и опубликуй один durable volume-capture для зафиксированного source UID».

Просьба живёт, пока не выполнена (`Ready`) или пока не доказано, что выполнить её нельзя (`Failed`). VCR **не знает про попытки**: сколько раз контроллер (и CSI-слой под ним) ходил в backend — это внутренняя кухня реализации request, а не изменение самого request.

### 1.1. Состояния

| Состояние | Смысл | Терминально? |
|---|---|---|
| `Progressing` | работа продолжается **или её исход неизвестен** | нет |
| `Ready` | durable `DataRef` опубликован | да (success) |
| `Failed` | доказано, что **просьба** не может быть выполнена | да (failure) |

`Stalled` — **отдельная condition, не фаза**: «давно нет наблюдаемого прогресса», не терминально, **без автоматических действий**. Оформляется как:

```yaml
conditions:
  - type: Ready
    status: "False"
    reason: CaptureInProgress
  - type: Stalled
    status: "True"
    reason: NoObservableProgress
```

`Stalled` намеренно не делается terminal-фазой, чтобы не путать «долго нет прогресса» с «request завершён».

### 1.2. Два разных вопроса (не смешивать)

- **«VCR невыполним?»** → решается по **request-level, apiserver-наблюдаемым** фактам источника. `Failed` описывает судьбу просьбы.
- **«Безопасно ли начать новую backend-операцию?»** → отдельный вопрос. `Failed` **не является** разрешением на cleanup/recreate.

Пример: source PVC удалён → просьба невыполнима → `Failed`; но неизвестно, успел ли прежний CSI-вызов создать артефакт → авто-начинать новую операцию всё равно не требуется.

## 2. Что контроллер имеет право делать автоматически

### 2.1. Разрешено

- Продолжать reconcile **существующего** исполнения: тот же `VolumeSnapshotContent`, тот же UID, та же CSI identity. Это и есть безопасный идемпотентный re-drive (он же уже происходит в CSI-сайдкаре ниже VCR).
- Восстанавливать **наблюдение** после рестартов контроллеров, конфликтов API, временной недоступности сайдкара.

### 2.2. Не разрешено без отдельного доказательства

- Удалить старый VSC, создать новый, начать новую CSI identity — **ни по timeout, ни по `Stalled`, ни просто потому, что VSC пропал из API.**

### 2.3. Terminal `Failed` — только по request-level фактам

Например: source UID больше не существует; source изменился и исходная identity невосстановима; режим принципиально не поддержан; отсутствует неизменяемая необходимая конфигурация; получен действительно нормативный terminal нижнего слоя. **Список нельзя строить по строкам `status.error`** — нужен надёжный канал классификации (см. §3.2, §3.3).

## 3. Проверено по коду (обоснование контракта)

Факты, подтверждённые в external-snapshotter (форк v8.5.0 в `storage-foundation`) и в контроллерах VCR/ObjectKeeper. Именно они закрывают две «опасные уверенности» прежней формулировки.

### 3.1. Пересоздание VSC ≠ та же backend identity — **подтверждено**

`CreateSnapshotRequest.Name` в VSC-only режиме (его и использует VCR) строится из **UID `VolumeSnapshotContent`**: `snapshot-<VSC.UID>`. Пересоздание VSC с тем же k8s-именем даёт **новый UID → новое имя backend-снапшота → новый физический снапшот** (в форке есть тест `TestVSCOnlyContract_RecreateVSC`, который прямо это фиксирует).

**Следствие:** утверждение «контент исчез → безопасно re-drive той же идентичности» **ложно**. Исчезновение API-объекта не доказывает, что физическая операция/снапшот исчезли из backend. Худший случай: VSC исчез из K8s, а CSI ещё создаёт `snapshot-UID-1`; создаём новый VSC → CSI начинает `snapshot-UID-2` → два физических снапшота.

### 3.2. `status.error` ненадёжен для машинной классификации — **подтверждено**

`VolumeSnapshotError` = `{time, message}`, **без** машинно-читаемого кода. Исходный gRPC-код в объекте не сохраняется. Классификация terminal по тексту `status.error` — ненадёжна.

Единственный машинный сигнал «сайдкар перестал повторять» — снятая аннотация `snapshot.storage.kubernetes.io/volumesnapshot-being-created` + появившийся `status.error`. Но «сайдкар-final» ≠ «просьба невыполнима»: драйвер мог вернуть final gRPC-код на транзиентную причину.

### 3.3. gRPC-классификация есть, но только в сайдкаре

`isCSIFinalError`: retryable = `Canceled/DeadlineExceeded/Unavailable/ResourceExhausted/Aborted` (+ не-gRPC → retryable «на всякий случай»); final = все остальные. Пока код retryable — сайдкар держит аннотацию being-created и повторяет `CreateSnapshot` (идемпотентно). Эта логика **не переносится** на объект VSC машинно-читаемо.

### 3.4. Окно удаления VSC — **важная оговорка (стык с delete-guard)**

Пока `CreateSnapshot` в полёте (аннотация being-created) — VSC из API не удалить (финализатор `volumesnapshotcontent-bound-protection` + аннотация держат). **Но** после успеха RPC и **до** `readyToUse=true` — удаление VSC **проходит** и триггерит удаление backend-снапшота. Значит managed VSC должен быть защищён delete-guard'ом **и в этом окне** (требование к предикату защиты VSC в delete-protection-контракте).

### 3.5. Безопасная форма «recovery» существует только одна

Пересоздать VSC с той же backend identity динамически **нельзя**. Единственный доказуемо-безопасный путь — **pre-provisioned rebind**: новый VSC с `spec.source.snapshotHandle` на существующий handle → сайдкар зовёт `GetSnapshotStatus/ListSnapshots`, а **не** `CreateSnapshot`. Это работает **только если** `snapshotHandle` был заранее сохранён (см. §6, open).

## 4. TTL — уже соответствует контракту

TTL не должен быть скрытым execution-deadline. Проверено, что код уже это соблюдает:

- **VCR-TTL** отсчитывается от `status.completionTimestamp`; non-terminal (Progressing) VCR **никогда** не удаляются по TTL. Долгий in-progress capture (часы/дни) TTL не оборвёт.
- **ObjectKeeper `FollowObjectWithTTL`** — TTL от удаления follow-объекта, а не от создания; фактический reconcile TTL — в deckhouse-controller (вне наших репо).

Контракт TTL: **срок хранения завершённого request, а не срок его выполнения** (`creation → Progressing/Stalled: TTL не применяется; Ready/Failed → completionTimestamp → retention TTL`). Явный `executionDeadline` возможен только как отдельная пользовательская политика, по умолчанию отсутствует, и даже при истечении **не** запускает новый backend-снапшот (максимум — `Failed`/operator-decision).

**Внешнее допущение:** корректность ObjectKeeper-TTL держится на deckhouse-controller; фиксируем как зависимость, не как нашу задачу.

## 5. Что остаётся от задачи C (не «retry», а long-lived execution contract)

1. VCR живёт до `Ready` или доказанного `Failed`.
2. TTL действует только после `completionTimestamp` (уже так в коде — §4).
3. Ошибка одной CSI-коммуникации **не** делает VCR terminal (расхождение с текущим кодом — §0).
4. `Stalled` — только диагностика, без авто-abandon/recreate.
5. Контроллер продолжает работать с **существующей** CSI identity.
6. Новая CSI identity автоматически не создаётся, пока нет доказанного безопасного протокола замены (§3.5).
7. `DataRef` — commit point; после него VCR не оживает.
8. Потеря артефакта после commit — `CaptureArtifactLost`, а не повторный capture.

**Расхождение с текущим кодом (суть работы C):** сегодня VCR one-shot и любая `VSC.status.error` → терминальный `SnapshotCreationFailed`. Контракт требует long-lived `Progressing`, отражающий нижний слой, с terminal только по request-level фактам, плюс `Stalled`-condition.

**Барьер не трогаем:** он уже ждёт логический результат (`SnapshotContent.status.data` покрыл target-UID'ы), а не конкретный VCR/UID. attempt-модель не вводится.

## 6. Открытые вопросы (до реализации)

- **Персистить ли `snapshotHandle`** (в `status.data`/на контенте) ради возможности безопасного pre-provisioned rebind-recovery (§3.5) — или сознательно не поддерживать авто-recovery и в спорных случаях всегда идти в operator-decision. От этого зависит, появится ли вообще recovery-путь.
- **Кто владеет классификацией** «request-level невыполнимость» — логично VCR-контроллер (единственный, кто читает CSI/VSC), но сигнал должен быть надёжнее строки `status.error` (§3.2–3.3).
- **Форма `Stalled`** — только по времени без прогресса; как оператор инициирует повтор (аннотация `RetryRequested` на VCR vs ручной delete+recreate). `Retain` на durable-контенте — страховка от дубля при ручном повторе.
- **Иммутабельность `spec.target`** — сейчас только в прозе CRD; возможный hardening (CEL) — отдельно.
