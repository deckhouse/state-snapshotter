# NamespaceSnapshot: aggregated manifests download (PR4)

**Назначение документа:** короткое объяснение для обсуждения на митинге. **Нормативный контракт** (endpoint, коды ошибок, алгоритм, циклы, дубликаты) — только в SSOT:

[`docs/state-snapshotter-rework/spec/namespace-snapshot-aggregated-manifests-pr4.md`](../docs/state-snapshotter-rework/spec/namespace-snapshot-aggregated-manifests-pr4.md).

Здесь — «человеческий» пересказ без противоречий с SSOT.

---

## 1. Зачем это нужно

**Уже есть:**

- **N2a** — рабочий снимок одного namespace и выгрузка **одного** `ManifestCheckpoint`.
- **N2b PR2/PR3** — дерево snapshot-ов как **структура** и агрегированный **Ready** у родителя.

**Проблема:** оператор не может одним запросом забрать **всё** состояние, если логический snapshot состоит из **нескольких** узлов (родитель + дети), у каждого из которых свой MCP.

**PR4 закрывает этот разрыв:** один HTTP-запрос → один JSON со всеми захваченными объектами по дереву.

---

## 2. Что мы хотим получить

**Сценарий:**

Запрос (полный путь — в SSOT §6; коротко для слайда):

```http
GET …/v1alpha1/namespaces/<ns>/namespacesnapshots/<name>/manifests
```

**Ответ при успехе:** один JSON-массив объектов Kubernetes (как при выгрузке одного MCP), но **склеенный** по всему поддереву:

```json
[
  { "...": "объекты из root MCP" },
  { "...": "объекты из child 1" },
  { "...": "объекты из child 2" }
]
```

Порядок элементов в массиве задан в SSOT: сначала объекты корневого MCP, затем поддеревья детей в порядке обхода (дети в графе — по **имени** узла).

---

## 3. Как это работает (интуитивно)

Дерево результата живёт на **`NamespaceSnapshotContent`**:

```text
NamespaceSnapshot (namespaced)
        ↓
status.boundSnapshotContentName → NamespaceSnapshotContent (root)
        ↓
status.childrenSnapshotContentRefs[]  →  дочерние NSC → …
```

У **каждого** узла NSC в статусе есть **`manifestCheckpointName`** → это уже существующий **`ManifestCheckpoint` + chunks**.

**Идея:** каждый узел = уже знакомый «кусок» манифестов; PR4 только **обходит граф** и **склеивает** куски через тот же **`ArchiveService`**, что и для одного MCP (без нового формата хранения и без прямого чтения chunks снаружи сервиса — см. SSOT §4).

**Важно:** для обхода **не** используются `childrenSnapshotRefs` на `NamespaceSnapshot` — только канонический граф на NSC (как в SSOT §2.2).

**Защита от цикла в графе:** если ссылки замкнулись, запрос целиком завершается ошибкой (детали — SSOT §3.2).

**Готовность узла (важно для митинга):** для выдачи манифестов **решающая** проверка — готовность **ManifestCheckpoint** внутри **`ArchiveService.GetArchiveFromCheckpoint`** (как у N2a single download). Условие **`Ready`** на **`NamespaceSnapshotContent`** в SSOT можно **дополнительно** трактовать как ранний **409**, **если** это совпадает с N2b design (§4.4 / §11), но **не** вместо проверки MCP. Подробности — SSOT §3.1 и §10.3.

---

## 4. Что делает PR4 одной фразой

**Собрать все MCP узлов дерева `NamespaceSnapshotContent` и отдать их содержимое одним JSON-массивом** (read-time, без предварительно сохранённого aggregated-артефакта).

---

## 5. Важное решение: fail-whole

Если **любой** узел или любой MCP в цепочке «не готов», не найден, бит при склейке, или нарушены правила целостности (цикл, дубликат объекта) → **весь запрос падает**, частичного тела ответа нет.

**Зачем так:**

- снимок трактуется как **целостное** состояние;
- partial-ответ сложно интерпретировать и поддерживать;
- это согласуется с **fail-closed** линией N2 в design (см. также partial policy в `namespace-snapshot-controller.md` §9).

Таблица кодов — в SSOT §7.

---

## 6. Почему именно так

### 6.1 Не делаем partial download

Иначе неочевидно, что именно отсутствует, сложнее отладка и ниже доверие к тому, что «это и есть snapshot».

### 6.2 Не вводим tar / bundle в PR4

Уже есть выдача **JSON** с MCP и **`ArchiveService`**. Минимальный шаг — **конкатенация массивов** на чтении (см. SSOT §5). При желании клиент может сам упаковать ответ.

### 6.3 Не делаем новый storage

Aggregated результат **не материализуем** в etcd/отдельном объекте: только чтение на лету → проще rollout и меньше мест для рассинхрона.

### 6.4 Дубликаты

**Не** делаем last-write-wins, merge и **тихую** дедупликацию: на митинге короткий ответ — **дубликат одного и того же** `(apiVersion, kind, namespace, name)` в aggregated manifests означает **неконсистентный graph/capture contract** (логическая модель snapshot «сломана»), а не обычную коллизию данных, которую можно «приглушить».

Если после склейки два элемента совпадают по этому ключу — **ошибка всего запроса** (**500**, SSOT §5.2 / §7.1).

---

## 7. Что пользователь получает

1. **Один запрос** — скачать логический namespace snapshot целиком (в пределах manifests-only дерева N2b).
2. **Только захваченное** — те же правила, что у MCP: не «live» состояние кластера, а артефакт capture.
3. **Сжатие:** при `Accept-Encoding: gzip` ответ может отдаваться в gzip, по тому же принципу, что и для single-MCP `/manifests` (SSOT §6.3).

---

## 8. Что намеренно не входит в PR4

- Частичная выгрузка поддерева (partial subtree).
- Семантика optional/required детей **для этого endpoint** (в SSOT явно вне scope).
- Data layer (PVC, VolumeSnapshot, …).
- Экспорт/импорт продуктового архива, отдельные сценарии restore.
- **Глобальная** пересортировка всего плоского массива объектов «в один канонический порядок» — вне PR4; фиксирован только **порядок обхода** дерева (родитель → поддеревья, дети по имени), см. SSOT §3.3 и §12.
- Отдельные глобальные лимиты на размер **всего** ответа по сумме узлов — в PR4 нет (есть только наследуемые per-MCP ограничения `ArchiveService`, см. SSOT §8).

---

## 9. Связь с PR2 / PR3

| PR | Смысл |
|----|--------|
| **PR2** | Появилось **дерево** (в т.ч. synthetic scaffold), refs на графе. |
| **PR3** | **Ready** родителя зависит от детей (агрегация состояния готовности). |
| **PR4** | **Читаем** то же дерево и отдаём **единый** manifests-ответ. |

Логически PR4 завершает manifests-only ось N2b до domain wiring (**PR5**).

---

## 10. Ограничения (честно)

- Большое дерево → **большой** ответ и нагрузка на память/сеть на стороне API (как и у крупного одиночного MCP).
- **Лимиты:** в PR4 **сознательно** нет **глобального** cap на суммарный размер ответа по всему subtree (SSOT §8.2) — это scope-cut для MVP; **per-MCP** ограничения `ArchiveService` (**`maxObjectsPerCheckpoint`**, предупреждение по размеру) **наследуются на каждый узел**. Отдельный hardening (aggregate limits, жёсткий потолок ответа) — следующий шаг после сквозного сценария.
- Один сбой в поддереве → **падает весь** запрос (by design).
- Дубликат объекта в склеенном массиве → **ошибка**, не silent merge.
- Цикл в ссылках → **ошибка** (не зацикливаемся молча).

---

## 11. Почему это приемлемо для MVP

- Цель PR4 — **сквозной** сценарий «забрал весь manifests-only snapshot» без новых сущностей в API.
- Усложнения (partial, streaming-протокол, data, продуктовый export) остаются **следующими этапами** (см. SSOT §12 и план N5).

---

## 12. Итог для слайда

**PR4** даёт оператору:

> «Скачай мне этот **NamespaceSnapshot** как **одно** JSON-состояние манифестов по всему дереву NSC.»

Без новых форматов хранения, без новых CRD под aggregated blob и без смены семантики **Ready** на существующих объектах.

## TL;DR

**PR4** — это: **обойти дерево `NamespaceSnapshotContent`, для каждого узла взять MCP через `ArchiveService`, склеить JSON-массивы в один ответ по фиксированным правилам; при любой проблеме — fail-whole.** Детали — только в [SSOT PR4](../docs/state-snapshotter-rework/spec/namespace-snapshot-aggregated-manifests-pr4.md).
